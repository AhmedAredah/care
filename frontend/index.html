Run the local pipeline against a folder of police crash report PDFs or scanned images. Files are read in place — originals are never modified, copied to the export directory, or sent off this machine.

Recent runs

What is a run?

A processing run is one batch invocation of the CARE pipeline: ingest → template detection → OCR/VLM extraction → PII detection → redaction → export. Each input file in the chosen folder produces one report. Reports whose extraction is uncertain are fail-closed: their redacted artifacts exist on disk but are flagged for human review before release.

What does each queue mean?

Needs attention — reports the pipeline either blocked (export refused by the QA gate) or marked requires_human_review (e.g. low template confidence, OCR/VLM disagreement, unmapped PII). Nothing here can be released publicly until you act on it.

Approved — reports a reviewer has signed off on. The redacted artifacts on disk are considered safe for release.

Rejected — reports a reviewer marked unsuitable. The redacted artifacts remain on disk for audit but should not be released.

      Reviewer actions

      What does approval do?

      Marking a report approved updates the in-memory review state only. CARE never re-runs redaction or unblocks an export the QA gate refused — you can still mark a report rejected after the gate let it through, or reject something the gate blocked. To regenerate artifacts, fix the upstream issue (template, OCR, PII chain) and start a new processing run.

      Exported artifacts

      Read-only listing of the redacted-report folders under the configured export_dir. Original PDFs, raw OCR/VLM dumps, and unredacted text are never exposed here — those are filtered server-side by the /api/exports endpoint.

      What lands in an exported folder?
      • diagram.redacted.png — pixel-redacted crash diagram crop
      • narrative.redacted.txt — placeholder-redacted officer's narrative (plain text)
      • narrative.redacted.json — same content with structured PII offsets
      • manifest.json — file SHA-256, template id, provider chain, model checksums
      • qa.json — QA flags, blocking reasons, confidence scores

      These five files are the complete public-safe export surface. Anything else under the report folder is a bug.

      Offline guard detail

      What does the offline guard do?

      The offline guard patches Python's socket and HTTP layers at startup so any code that tries to open an outbound connection raises OfflineGuardError and never sends a packet. It also sets the Hugging Face / Transformers offline environment variables (HF_HUB_OFFLINE=1, TRANSFORMERS_OFFLINE=1, …) so model libraries don't even attempt to phone home for cached metadata. 

      
    
      
  
      

  
  
      
    
      
      
      
        
      Plugins
      
        
      
          Every OCR engine, document-AI / VLM model, and PII detector in
          CARE is a plugin behind a stable interface. Operators choose
          which plugins are active; nothing is hard-coded into the core
          pipeline.
        
      
      
      
    
      
    
      
      How do plugin chains work?
      
      
        Each section (ocr, document_ai,
        pii) has a provider chain — an
        ordered list of plugin names. The pipeline runs them in order and
        merges results. Enabling a plugin both flips its
        enabled flag and appends it to the chain; disabling
        does the inverse. Per-provider config (model paths, thresholds,
        endpoints) lives on the Settings page.

      Template builder
      
        
      
          Author a state-and-version-specific template by drawing diagram
          and narrative regions on a sample report. Source PDFs are
          rendered into the configured work_dir only and are
          never copied to the public export directory. Saved templates are
          schema-validated before any bytes hit disk; existing files
          cannot be overwritten without explicit force=true.
        
      
      
      
    
      

    
      
      
      
        
      1. Load source PDF
      
        
      
          What kind of file should I use?
          
      
            A representative blank or synthetic-data sample of the form
            you're targeting (e.g. one filled-in example of a state's
            CR-3-equivalent crash report). Do not load
            real reports with PII — the builder renders pages to the work
            directory for visual inspection. Use the same template across
            all reports of that state-and-version.
          
      
        
      
      
      
      
      
        
        
        
      
          
          
        
      
      
      
      
      
    
      

    
      
      
      
        
      2. Mark identifiers and draw regions
      
        
      
          What's the difference between identifiers and anchors?
          
      
            Identifier words fingerprint the form so the
            pipeline can pick this template (e.g. "Crash Report",
            "MOSERS"). They must appear on every instance of
            this form.
            Narrative start/end anchors bound where the
            narrative paragraph begins and ends inside its page region —
            they are extraction guides, not template identifiers.

        manual anchor input below to type anchor strings
        instead. Region drawing still works.

            Suggestions never auto-apply — accept each one explicitly.
          
        
        
        
          
      
          
    
          

    
          
      
          
        
          
          
          3. Identify by
          
          
          
            What goes here?
            
          
              template_id is a short snake_case slug (e.g.
              example_state_crash_v1). jurisdiction
              is the state or agency code. version tracks the
              form revision. form_number_regex is optional —
              when present, the detector also matches the form number on
              the page (e.g. EX-CR-[0-9]+) to disambiguate
              very similar templates.
            
          
          
          
        
          
        
        
        
        
        
        
        
        
        
        
            
      
            

      
            
        
            
          
            4. Regions
            
          
            
            About diagram and narrative regions
            
            
              The diagram region tells the extractor where
              to crop the bird's-eye scene drawing. The
              narrative region bounds the text block where
              the narrative paragraph starts and ends. Both are stored as
              normalized [x0, y0, x1, y1] in the 0..1 range so
              they are page-size-independent.
            
            
          
            
        
            
        
            
          
            diagram
            
          
            (none)
            
          
          
            
          
            
            Advanced: diagram_continuation
            
            
              When the crash diagram spills onto a continuation page (rare
              but happens on multi-vehicle reports), this lets the
              extractor pick up the spill if visual density is high enough.
            
            
            
            
            
          
            
        
            
        
            
        
            
          
            narrative
            
          
            (none)
            
          
          
            
            
            
          
            
          
          
            
            
            
          
            
          
          
            
          
            
            Advanced: continuation
            
            
              For long narratives that wrap onto a continuation page.
            
            
            
            
            
            
            
            
            
            
          
            
          
            
            Advanced: shifted_region_search
            
            
              When the narrative box drifts off-position on real submissions
              (different scanner offset, slight form re-layout), this lets
              the extractor retry within nearby pages.
            
            
            
            
            
          
            
        
            
      
            
    
            

    
            
      
            
        
            5. Preview & save
            
        
            
          What does Preview do?
          
            
            Preview runs detect_template on the
            current draft against the loaded sample and returns the score —
            so you can confirm the identifiers and regions you picked are
            specific enough before committing the YAML to disk.
            Save validates against the template schema and
            writes templates/<jurisdiction>/<template_id>.yaml
            atomically. Existing files are never overwritten silently.
          
            
        
            
      
            
      
            
        
        
        
      
            
      
            
    
            
  
            

            



            
  
            
  
            
    
            
      
            New processing run
            
      
    
            
    
            
      
            
        
        
        
            
          A folder of .pdf, .png,
          .jpg, .jpeg, or .tiff
          files. Subdirectories are scanned recursively. Originals are
          read-only — CARE never modifies them.
        
            
      
            

      
            
        
        
            
          Default: auto-detect using every installed template.
          Restrict only when you know which form(s) are in this folder —
          a tighter allowlist reduces false-positive template matches.
        
            
        
            
          Restrict to jurisdiction or template ids
          
            
            Both filters are optional. When both are set, they apply
            with AND — a template must match the jurisdiction
            and be in the template-ids list. When neither is
            set, every installed template is in scope (auto-detect).
          
            

          
          
            
            What is jurisdiction?
            
            
              Jurisdiction is the state, DOT, or
              law-enforcement agency code declared on the template's
              jurisdiction: field — typically a state code
              (texas, missouri,
              iowa_dot) or a sub-agency abbreviation.
            
            
            
            
              What it matches: every installed
              template whose jurisdiction: field equals
              this value. Coarse filter. If Texas has three
              revisions of its CR-3 form installed
              (texas_crash_v1,
              texas_crash_v2,
              texas_crash_v3), they all match.
            
            
            
            
              When to use: the input folder is from
              one state but a mix of form revisions, or you don't know
              which revision was used.
            
            
            
            
              Why this exists: the federal MMUCC
              standard prescribes the data elements a crash
              report should capture, but each state designs its own
              police-crash-report form layout. CARE templates
              describe layouts, so this filter scopes the search to
              one state's set of layouts.
            
            
          
            
          
          
            
            Single state / agency code. Matches every template whose
            jurisdiction: field equals this value.
          
            

          
          
            
            What are template ids?
            
            
              Template ids are the unique slugs for
              specific form revisions (e.g.
              texas_crash_v2). Every installed template
              YAML carries a template_id: field; this
              filter matches only the listed ids.
            
            
            
            
              What it matches: only templates whose
              template_id: field appears in your list.
              Fine filter. Listing
              texas_crash_v2 excludes
              texas_crash_v1 even though both are Texas
              templates.
            
            
            
            
              When to use: a state issued a new
              revision but legacy reports under the old revision are
              still arriving, and you want each batch processed under
              the correct template; or the same agency uses
              different forms for different report types (e.g. a
              short-form vs. long-form crash report) and you want to
              constrain to one.
            
            
            
            
              Naming convention:
              <jurisdiction>_<form>_v<n>.
              The Template builder lowercases and snake-cases ids
              automatically; comma-separate multiple values here.
            
            
          
            
          
          
            
            One or more template IDs, comma-separated. Matches only
            templates whose template_id: field is in this
            list.
